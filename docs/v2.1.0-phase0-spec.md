## 1. 背景与目标

### 1.1 立项依据

v2.0.0 已完成 68 个任务（v0.2.0-v2.0.0 全量优化），496 pytest 全绿。对比分析显示三大战略差距：
- **G4（战略级）**: NomiFun 自建 Rust in-process Computer Use，无 Playwright 依赖
- **G12**: 桌面壳落后 2 代（PyWebView vs Tauri 2）
- **体积**: 54MB vs Tauri 原生 50-80MB

Rust 骨架模块已编译通过（browser_use + computer_use，PyO3 0.22），触发条件已满足。

### 1.2 核心目标

**重新定义为"架构重构"**（非性能优化），收益预期：
- ✅ **自动更新能力**: 内置 `tauri-plugin-updater`，对接 GitHub Releases
- ✅ **安全 posture 提升**: Tauri capabilities deny-by-default + CSP 强制 + 签名校验
- ✅ **渐进 Rust 迁移基础**: 铺设 Phase 1-3 的技术路径
- ✅ **跨平台分发**: MSI/DMG/AppImage 原生安装包

### 1.3 性能预期修正（基于七位专家评测）

| 指标 | 原预期 | 修正值 | 说明 |
|------|--------|--------|------|
| 包体 | 15-20MB | **48-67MB** | Python 依赖栈下限 34-51MB，Phase 0 不减总量 |
| 启动(窗口可见) | <800ms | <500ms ✓ | Tauri 壳秒开 |
| 启动(功能可用) | <800ms | **1.5-3s** | Python sidecar 启动不变 |
| 内存 | 100-180MB | **190-260MB** | 双进程叠加可能反升 |
| IPC p95 | 未列 | 2-8ms | invoke 中转会增加延迟 |

**关键认知**：性能收益要等 Phase 4（移除 Python）才能真正兑现。Phase 0 不以性能为立项理由。

### 1.4 非目标

- ❌ 全面 Rust 重写（14.5 人月，方案 A）
- ❌ 性能优化（包体/启动/内存）
- ❌ 业务逻辑迁移（Phase 2-3）
- ❌ 数据库迁移（保持 SQLAlchemy）
- ❌ 前端框架替换（保持 Preact）

## 2. 架构设计

### 2.1 双进程模型

```
┌──────────────────────────────────────────────────────────┐
│  Tauri 2 主进程（Rust）                                    │
│  ├─ WebviewWindow（加载 Preact 前端）                      │
│  ├─ TrayIconBuilder（托盘 + 单实例锁）                     │
│  ├─ tauri-plugin-updater（自动更新 + 签名校验）             │
│  ├─ Sidecar Supervisor（Python 进程生命周期管理）           │
│  │   ├─ 启动 Python sidecar（动态端口）                    │
│  │   ├─ 健康检查轮询（/health/ready）                      │
│  │   ├─ 崩溃检测 + 指数退避重启（上限 3 次）                │
│  │   └─ 优雅关闭（SIGTERM → 等待 5s → kill）              │
│  ├─ Sidecar 完整性校验（SHA-256 哈希清单）                 │
│  └─ IPC 代理层（http_proxy command）                      │
└──────────────────────────────────────────────────────────┘
                          ↕ HTTP localhost（动态端口）
┌──────────────────────────────────────────────────────────┐
│  Python Sidecar（FastAPI 子进程）                          │
│  ├─ 333 个 API 端点（零改动）                              │
│  ├─ 68 个服务模块（零改动）                                │
│  ├─ SQLAlchemy + aiosqlite（零迁移）                      │
│  ├─ DPAPI / E2EE / CRDT（保留）                           │
│  └─ 启动时 stdout 输出 PORT=xxxxx                         │
└──────────────────────────────────────────────────────────┘
```

### 2.2 IPC 通信策略（混合分流）

**核心原则**：不一刀切，按场景分流。

| 场景 | 通信方式 | 理由 |
|------|---------|------|
| 普通 CRUD（apiGet/apiPost/apiPut/apiDelete） | Tauri invoke → Rust reqwest → Python | 权限管控 + 统一错误处理 |
| **SSE 流式（apiStream / ChatPanel）** | **前端直连 Python HTTP** | invoke 不支持流式，保留直连避免破坏 |
| 终端轮询（TerminalPanel 800ms） | Tauri invoke | 低频，可走代理 |
| 健康检查（StatusBar 10s） | 前端直连 Python | 极低频，避免 Tauri 中转 |
| 文件操作 / 系统调用 | Tauri 原生 command | 不经 Python，直接 Rust |

**关键决策**：`apiStream` 显式排除在 invoke 之外，保留 fetch + ReadableStream 直连。

### 2.3 端口协商协议

```
1. Tauri 启动 → spawn Python sidecar（无参数）
2. Python sidecar: bind(("127.0.0.1", 0)) → OS 分配端口
3. Python sidecar: stdout 输出 "PORT=xxxxx\n"
4. Tauri: 读取 stdout，解析 PORT
5. Tauri: 通过 window.__NEBULA_PORT__ 注入前端
6. 前端 lib/api.ts: 动态读取 API_BASE
7. Tauri: 轮询 http://127.0.0.1:PORT/health/ready，就绪后显示主窗口
```

### 2.4 Sidecar 生命周期管理

| 阶段 | 实现 |
|------|------|
| **启动** | Tauri `setup` 钩子 spawn sidecar，CSP 允许 `http://127.0.0.1:*` |
| **就绪检测** | 新增 `/health/ready` 端点（返回 DB 初始化状态），Tauri 轮询 |
| **崩溃检测** | `CommandEvent::Terminated` 监听 + `/health/ready` 主动轮询（5s 间隔） |
| **重启策略** | 指数退避（1s→2s→4s），上限 3 次，超限显示降级 UI |
| **优雅关闭** | 新增 `/shutdown` 端点，Tauri 退出前调用 → 等待 5s → kill |
| **完整性校验** | 构建时生成 sidecar.sha256，Tauri 启动前校验 |

### 2.5 Feature Flag 回退机制

```python
# launch.py 保留作为 PyWebView 回退入口
import os
SHELL = os.environ.get("NEBULA_SHELL", "pywebview")  # tauri | pywebview
if SHELL == "tauri":
    # Tauri 模式：仅启动 Python sidecar，不开窗口
    run_sidecar_only()
else:
    # PyWebView 模式：原有逻辑
    run_pywebview()
```

### 2.6 Rust 模块双目标改造

```toml
# rust/browser_use/Cargo.toml + rust/computer_use/Cargo.toml
[lib]
crate-type = ["cdylib", "rlib"]  # ← 同时支持 Python 扩展和 Rust 库

[features]
default = []
python = ["dep:pyo3"]  # ← 仅 Python 目标启用 pyo3

[dependencies]
pyo3 = { version = "0.22", features = ["extension-module"], optional = true }
```

Phase 0 改造（即使逻辑仍是骨架），为 Phase 1 Tauri 直调铺路。

## 4. 安全设计

### 4.1 信任模型

- **Rust 侧（信任根）**: 负责 sidecar 生命周期、完整性校验、IPC 路由
- **Python sidecar（半可信）**: 处理业务逻辑，无权直接调 OS 原生能力
- **数据校验**: Rust 侧对 sidecar 返回数据 serde 反序列化校验

### 4.2 P0 安全加固（启动前必须完成）

| # | 措施 | 实现 |
|---|------|------|
| 1 | Sidecar 完整性校验 | 构建时生成 sidecar.sha256，Tauri 启动前校验 |
| 2 | IPC 链路认证 | 启动时 sidecar 生成随机 token → stdout → Tauri → Bearer token |
| 3 | CORS 收紧 | 移除 `http://127.0.0.1:*`，仅 `tauri://*` / `app://*` |
| 4 | cargo audit 纳入 CI | 新增 rust-audit job |
| 5 | Sidecar 更新签名 | sidecar 哈希清单纳入 updater 签名范围 |

### 4.3 Tauri Capabilities 配置

**deny-by-default 原则**：
- 每窗口独立 capability 文件
- 仅显式列出的命令可被前端调用
- `fs:scope` 限制到 `$APPDATA/nebula/**`
- `shell:allow-execute` 默认 deny（终端功能按需开启 + 白名单）
- CI lint 检查无 `*` 通配符

### 4.4 现有安全措施迁移状态

| # | 措施 | 状态 | 说明 |
|:-:|------|:----:|------|
| 1 | DPAPI 凭据保护 | ✅ 保留 | sidecar 同用户运行 |
| 2 | AES-256-GCM 加密 | ✅ 保留 | 逻辑在 sidecar |
| 3 | ACL 访问控制 | ⚠️ 适配 | 扩展至 invoke 命令层 |
| 4 | 注入防护 | ⚠️ 适配 | Rust 入口 + Python 入口双校验 |
| 5 | SSRF 防护 | ✅ 保留 | sidecar 出站仍走此模块 |
| 6 | 沙箱执行 | ❌ 重设计 | Phase 1 升级为 OS 级沙箱 |
| 7 | 审计日志 | ⚠️ 扩展 | 新增 Tauri 侧事件审计 |
| 8 | 密钥轮换 | ✅ 保留 | 逻辑在 sidecar |
| 9 | OAuth 2.0 | ✅ 保留 | 逻辑在 sidecar |
| 10 | DID | ✅ 保留 | 逻辑在 sidecar |
| 11 | E2EE + 脱敏 | ⚠️ 适配 | redactor 覆盖 Tauri 侧日志 |

### 6.2 lib/api.ts 双实现

```typescript
const isTauri = '__TAURI_INTERNALS__' in window
const API_BASE = isTauri
  ? `http://127.0.0.1:${(window as any).__NEBULA_PORT__}`
  : 'http://127.0.0.1:7860'

async function request<T>(method, path, body?): Promise<T> {
  if (isTauri) {
    // Tauri: invoke http_proxy command
    const json = await invoke('http_proxy', { method, path, body })
    if (!json.ok) throw new Error(json.error)
    return json.data
  } else {
    // 浏览器: 保留原 fetch（脱离 Tauri 独立开发）
    const res = await fetch(`${API_BASE}${path}`, {...})
    const json = await res.json()
    if (!json.ok) throw new Error(json.error)
    return json.data
  }
}

// apiStream 保留直连 fetch（不走 invoke）
export async function apiStream(path, body, onChunk) {
  const res = await fetch(`${API_BASE}${path}`, { method: 'POST', ... })
  // 原有 ReadableStream 解析逻辑不变
}
```

### 7.1 风险矩阵

| ID | 风险 | 概率 | 影响 | 缓解 |
|----|------|:----:|:----:|------|
| R1 | 工作量超估 | 90% | 高 | 工期修正为 2 月 |
| R2 | Rust 学习曲线 | 85% | 高 | 启动前 2 周培训 + 外部 mentor |
| R3 | Sidecar 集成坑 | 70% | 高 | Week 2 做 PoC 硬检查点 |
| R4 | 单点故障 | 50% | 极高 | 强制 2 人结对 + 文档化 |
| R5 | 打包跨平台回归 | 75% | 中 | 提前申请签名证书 |
| R6 | 业务连续性中断 | 80% | 中 | 维护 maint/v2.0.x 分支 |

| R8 | WebKitGTK 版本 | 100% | 高 | 立即修正 CI 4.0→4.1 |

### 7.2 Go/No-Go 检查点

| 检查点 | 时间 | 标准 | 失败动作 |
|--------|------|------|---------|
| 启动前置 | Week 0 | 证书申请 + Rust 培训 + feature flag | 延期启动 |
| 脚手架 | Week 1 | `cargo tauri dev` 可启动 | 延长 1 周 |
| **Sidecar PoC** | **Week 2** | **Tauri 启动 Python sidecar + /health 200** | **触发回退讨论** |
| 核心功能 | Week 4 | 仪表盘/记忆图/技能市场可加载 | 评估延期 |
| 打包 | Week 6 | Windows MSI 可安装 | 延期或降级 |
| 回归 | Week 8 | 496 pytest 全绿 | 阻断发布 |

### 7.3 回退方案

1. **Feature flag**: `NEBULA_SHELL=tauri|pywebview` 环境变量控制
2. **launch.py 保留**: 不删除原 PyWebView 入口
3. **lib/api.ts 双实现**: 保留 fetch 后备，支持脱离 Tauri 开发
4. **release.yml 保留**: PyInstaller 路线作为 fallback，Tauri 稳定 3 次发布后废弃
5. **数据零迁移**: SQLite 格式不变，回退无数据成本
